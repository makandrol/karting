# Технічна документація: Визначення груп, пілотів та прив'язка заїздів

## Огляд

Коли починається змагання, система повинна:
1. Визначити кількість пілотів
2. Розбити пілотів на групи
3. Прив'язати кожний заїзд до відповідної фази змагання (кваліфікація, гонка, фінал)

Ці процеси відбуваються як автоматично (через колектор), так і вручну (через адмін-панель).

---

## 1. Створення змагання

### Автоматичне (через розклад)

Колектор перевіряє розклад (`collector/src/schedule.js`):

| День      | Формат             | Вікно детекції |
|-----------|--------------------|----------------|
| Понеділок | Гонзалес           | 19:30–20:30    |
| Вівторок  | Лайт Ліга          | визначено      |
| Середа    | Ліга Чемпіонів     | визначено      |

`CompetitionDetector` (`collector/src/detector.js`) переходить через стани:
```
NONE → WARMUP → QUALIFYING/RACE → FINISHED
```

Перехід `WARMUP → QUALIFYING` відбувається коли між заїздами виникає пауза ≥3 хвилин (зміна траси для змагань).

### Ручне (через адмін-панель)

Адмін створює змагання через `CompetitionManager` — POST `/competitions` з полями:
- `id`: `{формат}-{дата}-{timestamp}`
- `name`, `format`, `date`
- `sessions: []` (порожній масив, заповнюється потім)
- `status: 'live'`

---

## 2. Прив'язка заїздів до фаз

### Фази кожного формату

Кожний формат має фіксований впорядкований список фаз. Заїзди прив'язуються **послідовно** — перший заїзд отримує першу фазу, другий — другу і т.д.

**Лайт Ліга** (макс. 3 групи, 2 гонки):
```
qualifying_1 → qualifying_2 → qualifying_3 → qualifying_4
→ race_1_group_3 → race_1_group_2 → race_1_group_1
→ race_2_group_3 → race_2_group_2 → race_2_group_1
```

**Ліга Чемпіонів** (макс. 2 групи, 3 гонки):
```
qualifying_1 → qualifying_2
→ race_1_group_2 → race_1_group_1
→ race_2_group_2 → race_2_group_1
→ race_3_group_2 → race_3_group_1
```

**Спринт** (макс. 3 групи, 2 гонки + фінал):
```
qualifying_1_group_1 → qualifying_1_group_2 → qualifying_1_group_3
→ race_1_group_3 → race_1_group_2 → race_1_group_1
→ qualifying_2_group_1 → qualifying_2_group_2 → qualifying_2_group_3
→ race_2_group_3 → race_2_group_2 → race_2_group_1
→ final_group_3 → final_group_2 → final_group_1
```

**Гонзалес** (макс. 2 групи, 12–24 раунди):
```
qualifying_1 → qualifying_2
→ round_1_group_2 → round_1_group_1
→ round_2_group_2 → round_2_group_1
→ ... → round_12_group_1
```

### Фільтрація фаз по кількості груп

Якщо визначено, що груп лише 2 — всі фази з `group_3` видаляються зі списку. Якщо 1 група — видаляються `group_2` і `group_3`.

Наприклад, Лайт Ліга з 2 групами:
```
qualifying_1 → qualifying_2
→ race_1_group_2 → race_1_group_1
→ race_2_group_2 → race_2_group_1
```

### Порядок гонок всередині раунду

Слабші групи їдуть першими: group_3 → group_2 → group_1 (найсильніша група завжди остання).

### Механізм авто-прив'язки

Коли починається новий заїзд, колектор (`poller.js`) викликає `storage.autoLinkSessionToActiveCompetition(sessionId)`:

1. Знаходить **перше** змагання зі статусом `live`
2. Визначає `groupCount` (пріоритети нижче)
3. Фільтрує повний список фаз по `groupCount`
4. Знаходить **наступну** незайняту фазу
5. Додає `{ sessionId, phase }` до `sessions[]` змагання

**Захист від переповнення**: якщо всі фази вже зайняті — нові заїзди не прив'язуються (запобігає випадковому захопленню прокатів після завершення змагання).

### Авто-відв'язка коротких заїздів

Заїзди коротші за 60 секунд автоматично відв'язуються — це помилкові/тестові заїзди.

---

## 3. Визначення кількості пілотів

### Відкуди береться число пілотів

1. **З кваліфікаційних заїздів**: `COUNT(DISTINCT pilot)` серед валідних кіл (≥38 секунд)
2. **Обрізка за форматом**: макс. 36 (ЛЛ), 24 (ЛЧ), 45 (Спринт), 24 (Гонзалес)
3. **Ручний оверрайд**: адмін може встановити `totalPilotsOverride` + `totalPilotsLocked`

### Пріоритет

```
totalPilotsOverride (ручний) → autoTotalPilots (з квали)
```

Якщо `totalPilotsLocked === true`, використовується `totalPilotsOverride` незалежно від реальних даних.

### Пілоти, що не пройшли кваліфікацію

Якщо пілотів більше за максимум формату — зайві (з найгіршим часом кваліфікації) отримують статус "дискваліфіковані" — відображаються в таблиці, але з 0 балів.

---

## 4. Визначення кількості груп (groupCount)

`groupCount` — центральне значення, що визначає скільки груп пілотів буде у змаганні. Від нього залежить список фаз і розбиття пілотів.

### Ланцюг пріоритетів

```
groupCountOverride (ручний від адміна)
  → autoDetectedGroups (автовизначення колектором)
    → autoGroups (автопідрахунок фронтендом з кваліфікацій)
```

### Автовизначення колектором (overlap-аналіз)

Після початку нового заїзду, колектор порівнює пілотів нового заїзду з кумулятивним списком пілотів з усіх попередніх кваліфікацій:

**Для ЛЛ/ЛЧ** (не Гонзалес):
- Якщо ≥50% пілотів нового заїзду вже були в попередніх кваліфікаціях → це **гонка**, а не ще одна кваліфікація
- `groupCount = кількість кваліфікаційних заїздів` (кожна кваліфікація = 1 група)
- Приклад: 2 квалі + новий заїзд з тими ж пілотами → `groupCount = 2`

**Для Гонзалес**:
- Перевіряє, чи імена пілотів реальні (не "Карт N")
- Реальні імена + мало кіл → це ще одна кваліфікація
- Імена як "Карт 1", "Карт 2" + багато кіл (≥5) → це раунди

**Для Спринт**:
- Підраховує кількість різних `qualifying_1_group_N` фаз з даними

### Автопідрахунок фронтендом

Якщо колектор не визначив `groupCount`:
- **ЛЛ/ЛЧ**: `groupCount = кількість кваліфікаційних сесій з даними` (макс за форматом)
- **Спринт**: `groupCount = кількість різних group_N у qualifying_1` фазах

### Формули групування за кількістю пілотів

| Формат          | ≤13 | 14–24/26/29 | 27+/30+ |
|-----------------|-----|-------------|---------|
| Лайт Ліга       | 1 гр. | 2 гр. (14–26) | 3 гр. (27–36) |
| Ліга Чемпіонів  | 1 гр. | 2 гр. (14–24) | макс. 2 |
| Спринт          | 1 гр. | 2 гр. (15–29) | 3 гр. (30–45) |
| Гонзалес        | 1 гр. | 2 гр. (14–24) | макс. 2 |

### Перерахунок фаз при зміні groupCount

Коли адмін змінює `groupCountOverride`, всі існуючі заїзди **перерозподіляються** по новому списку фаз (в тому ж порядку). Якщо нових фаз більше — система спробує знайти й прив'язати додаткові вільні заїзди з того ж дня.

## 5. Розбиття пілотів на групи

### ЛЛ / ЛЧ — Послідовне розбиття (`splitIntoGroups`)

Пілоти сортуються за часом кваліфікації і розбиваються **суцільними блоками**:
```
Пілоти: [1, 2, 3, ..., 24] (відсортовані за квалі)
2 групи → Група A: [1–12], Група B: [13–24]
3 групи → Група A: [1–12], Група B: [13–24], Група C: [25–36]
```

При нерівній кількості — перші (сильніші) групи отримують більше пілотів:
```
25 пілотів, 2 групи → Група A: 13 пілотів, Група B: 12 пілотів
```

### Спринт — Snake/Round-Robin розбиття (`splitIntoGroupsSprint`)

Для гонок 1 та 2 використовується **змійка** для балансування сил:
```
Пілоти: [1, 2, 3, 4, 5, 6, ...] (відсортовані за квалі)
3 групи, змійка:
  Група A: 1, 4, 7, 10, ...
  Група B: 2, 5, 8, 11, ...
  Група C: 3, 6, 9, 12, ...
```

Якщо кількість пілотів не ділиться порівну — розподіл реверсується, щоб **остання група** (яка їде першою) отримала більше пілотів.

**Для фіналу** — повертається до послідовного ярусного розбиття:
- Пілоти сортуються за сумою балів Г1+Г2
- Pro (група 1) = найкращі, Gold (група 2) = середні, Light (група 3) = решта

### Гонзалес — ротація картів

Немає класичних груп. Кожний пілот отримує стартовий слот і ротується через всі 12 картів. Якщо пілотів більше за 12, деякі слоти — "пропуски" (пілот не їде цей раунд).

---

## 6. Перевірка фази після першого кола (recheckSessionPhase)

Після того як у новому заїзді з'являється перше коло, колектор запускає `recheckSessionPhase`:

1. Збирає пілотів нового заїзду
2. Збирає кумулятивних пілотів з усіх попередніх кваліфікацій
3. Рахує % перетину

**Результат**:
- Якщо ≥50% перетин → це гонка (фаза `race_*`) → при потребі перепризначає фазу
- Якщо <50% перетин → це нова кваліфікаційна група

Для Гонзалес додаткова логіка: перевіряє формат імен пілотів ("Карт N" = раунди, реальні імена = кваліфікація).

## 8. Життєвий цикл змагання

```
Створення (live) → Авто-прив'язка заїздів → Перевірка фаз →
→ Всі фази заповнені → Адмін завершує (finished)
```

1. **Створення**: статус `live`, заїзди `[]`
2. **Під час змагання**: кожний новий заїзд автоматично отримує наступну фазу
3. **Рекалібрація**: після першого кола перевіряється правильність фази
4. **Короткі заїзди** (<60с): автоматично відв'язуються
5. **Захист від переповнення**: коли всі фази зайняті, нові заїзди ігноруються
6. **Завершення**: адмін змінює статус на `finished`
