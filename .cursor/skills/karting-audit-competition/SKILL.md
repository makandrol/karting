# Audit Competition vs Official Sheet

Перевіряє завершене змагання (Лайт Ліга / Ліга Чемпіонів) проти офіційної Google-таблиці, застосовує узгоджені автоматичні корекції і видає чіткий звіт про те, що ще не сходиться (баги логіки) і що було змінено вручну.

Усе працює через `scripts/audit/` (плейн-TS, запуск `npx tsx`), на справжній frontend-логіці `computeStandings` (без браузера). Пише в колектор через HTTP API (прод `141.147.32.196:3001`, ADMIN_TOKEN порожній → write відкритий).

## Передумови

- `tsx` встановлено (`npm install -D tsx` якщо нема).
- Колектор доступний (`COLLECTOR_URL`, дефолт прод).
- **Завжди спершу DRY-RUN, показати користувачу, потім `--apply`.**
- Перед першим `--apply` на сесії переконайся що є бекап БД. Роби дамп на VPS і **завжди лишай тільки 2 найновіші бекапи** (решту видаляй перед створенням нового — БД ~1GB, забиває пам'ять):
  ```bash
  ssh -i ~/.ssh/id_github ubuntu@141.147.32.196 "cd /home/ubuntu/collector/data && cp karting.db karting.db.bak-$(date +%Y%m%d-%H%M%S) && ls -1t karting.db.bak-* | tail -n +3 | xargs -r rm -f"
  ```

## Один прохід (головна команда)

**DRY-RUN — це режим за замовчуванням.** Без `--apply` скіл НІЧОГО не пише — лише читає дані й друкує звіт. Запускай dry-run скільки завгодно разів безпечно.

```bash
# DRY-RUN (дефолт — нічого не пише, лише звіт):
npx tsx scripts/audit/audit-competition.ts <competitionId>

# DRY-RUN з перестворенням (показує план лінкування, але не пише):
npx tsx scripts/audit/audit-competition.ts <competitionId> --recreate

# Застосувати корекції (штрафи + повернені фініші + траса + журнал змін):
npx tsx scripts/audit/audit-competition.ts <competitionId> --apply

# Застосувати з перестворенням з першого заїзду:
npx tsx scripts/audit/audit-competition.ts <competitionId> --apply --recreate

# З явним gid/url таблиці (якщо авто-резолв не знає вкладку):
npx tsx scripts/audit/audit-competition.ts <competitionId> <gidOrUrl> --apply
```

Опції:
- `--apply` — записати зміни (без нього — dry-run).
- `--recreate` — видалити + перестворити змагання з першого заїзду (валідує лінкування). Дефолт: НІ.
- `--max=N` — максимальна різниця для імпорту фінішної корекції (дефолт 3).
- `--name="..."` / `--date=YYYY-MM-DD` — лише з `--recreate`, якщо треба виправити назву/дату (буває, що дата змагання збита, напр. LL 28.04 збережене як 29.04).

Таблиця авто-резолвиться за форматом + датою першої сесії (мапи вкладок `LL_TABS`/`CL_TABS` у `scripts/audit/lib.ts`). CL за червень — окрема книга, передавай url явно.

## Повний flow «аудит змагання Х з перестворенням» (роби саме так)

Коли користувач каже "зроби аудит змагання Х **з перестворенням**" — виконуй цей порядок:

1. **Знайди competitionId.** Якщо дано URL/дату — витягни id. Інакше знайди за форматом+датою:
   ```bash
   curl -s "http://141.147.32.196:3001/competitions" | npx tsx -e "let d='';process.stdin.on('data',c=>d+=c).on('end',()=>{const j=JSON.parse(d);const a=Array.isArray(j)?j:[];for(const c of a.filter(x=>x.format==='light_league').sort((x,y)=>(x.date||'').localeCompare(y.date||'')))console.log(c.date,'|',c.id,'|',c.name);})"
   ```
   ⚠️ **Дата в id може відставати на 1 день** від назви (напр. «27.05» → сесії під UTC-датою `2026-05-26`, бо старт 19:49 Kyiv). Це нормально.

2. **Backup БД** (лишаючи 2 останніх — команда вище). Обов'язково перед `--apply`.

3. **DRY-RUN recreate** — подивись структуру та `Replay fromTs`:
   ```bash
   npx tsx scripts/audit/recreate.ts <id>
   ```
   Перевір: `Replay fromTs` має вказувати на ПЕРШУ валідну квалу (19:45-ish), НЕ на пізнішу. Якщо `⚠️ comp.date=... but first session is on ...` — заплануй передати `--date=` + `--name=` у apply, щоб зберегти назву/id.

4. **Apply recreate + аудит** (одна команда — оркестратор робить recreate → track → penalties → finishes → editlog):
   ```bash
   npx tsx scripts/audit/audit-competition.ts <id> --recreate --apply
   # якщо дата в id відстає — зберегти назву/дату:
   #   спершу: npx tsx scripts/audit/recreate.ts <id> --apply --date=YYYY-MM-DD --name="ЛЛ, DD.MM.26, Тр. N"
   #   потім:  npx tsx scripts/audit/audit-competition.ts <id> --apply
   ```
   Очікуй `RESULT — collector linked N sessions` + `STRUCTURE vs BEFORE`. **Перевір структуру:** LL = квалі (3-4) + 6 гонок (3 групи × 2), порядок `race_1_group_3→2→1`, `race_2_group_3→2→1`. Якщо якась гонка/квала DROPPED або відсутня `race_X_group_Y` — розслідуй (див. нижче).
   - ⚠️ `fetch failed`/socket на САМОМУ КІНЦІ (фінальний звіт) — це вже ПІСЛЯ запису; recreate+корекції застосовані. Просто перезапусти dry-run щоб побачити звіт.

5. **Прочистка «Карт N» у квалі більше НЕ потрібна вручну** — `computeStandings` тепер сам зливає їх через `mergePilotNames` (v0.9.494). Якщо все ж бачиш пілота як голе прізвище / «Карт N» у «Лише в нас» — це або справжній прогрів (0 валідних кіл), або reset lap_number (інший водій) — тоді лишаємо.

6. **Прочитай фінальний звіт** (нижче) і покажи користувачу таблицю балів + стартові розбіжності.

Одна команда `audit-competition.ts <id> --recreate --apply` покриває майже все. Окремі скрипти — лише для точкового дебагу.

## Що робить оркестратор (по кроках)

1. **(опц.) recreate** — `recreate.ts`: delete → create → авто-лінк сесій з першого заїзду (та сама логіка, що `SessionTypeChanger`). Зберігає `trackId`, `racePilotCount`, `excludedPilots`, `excludedLaps`, `edits`, `editLog`. Дату бере з ПЕРШОЇ СЕСІЇ (не з `comp.date`).
2. **track** — читає конфігурацію траси із заголовка таблиці ("(конфігурація N)" / "NR" → trackId N або N+100 для реверсу). Правило:
   - у нас траси НЕМАЄ (`trackId` порожній) → ставить з таблиці (+ `update-track` для сесій);
   - є і ЗБІГАЄТЬСЯ → нічого;
   - є і ВІДРІЗНЯЄТЬСЯ → **НЕ змінює**, лише попереджає (розбираємось вручну).
3. **import-penalties** — штрафи з таблиці → `results.edits` (`pilot|race` → `{penalties}`). Таблиця = джерело правди для штрафів.
4. **import-finishes** — фінішні корекції ТІЛЬКИ якщо `sheetFinish < ourFinish` (повернули позицію за фол) і різниця `<= --max`. Якщо таблиця ОПУСТИЛА (sheet > ours) — лишаємо наше.
5. **rebuild-editlog** — відновлює `results.editLog` з `edits`, щоб сторінка "Журнал змін" показувала всі правки (інакше скрипти оминають журнал).
6. **Фінальний звіт** — три секції (див. нижче).

## Звіт у кінці

- **⚠️ СТАРТОВІ ПОЗИЦІЇ** — список розбіжностей старту (наш vs табл). Плюс детект дублів/пропусків у стартових позиціях таблиці.
- **◆ БАЛИ — всі пілоти** — таблиця `Пілот | Бали | Табл | Δ | ✓/✗` (сортування за нашими балами) + підсумок `Збігається: N/M`. Головний індикатор коректності.
- **── ГОНКА K, старт** (друкується лише коли є стартові розбіжності) — по одній таблиці на гонку, згруповано за **групою ТАБЛИЦІ**, сортування старту з останньої позиції до 1 (щоб при правильному реверсі час кола йшов монотонно). Колонки: `Пілот | гр-табл | гр-наш | стрт-табл | стрт-наш | ✓ | джерело`. Джерело старту: **Гонка 1 → час кола квалі**; **Гонка 2+ → група і час кола попередньої гонки**. Ці таблиці одразу показують ПРИЧИНУ розбіжності старту (різна група / зсув / реверс).
- **◆ БАЛИ / ФІНІШІ** — залишкові розбіжності балів із причиною (часто "таблиця опустила — лишаємо наше").
- **✎ РУЧНІ ЗМІНИ** — повний список застосованих `edits` (старт / фініш / штраф / група).

### Як читати стартові розбіжності (діагностика по джерелу)

- **гр-табл ≠ гр-наш** (пілот в іншій групі) → розподіл по групах розійшовся. Причина майже завжди — **неправильний best-lap квалі** (див. «Карт N» нижче) або граничний пілот на межі груп. Звір колонку «Кв-час»: групи в таблиці = чистий best-lap (як наш `splitIntoGroups`), лише зрідка організатор вручну пересаджує когось (тоді — `edit.group`).
- **гр однакова, стрт зсув на +1** у всій групі → у нашій групі є ЗАЙВИЙ пілот (no-show, що кваліфікувався але не гонявся; або пілот, якого таблиця перекинула в іншу групу) — він займає слот і зсуває решітку.
- **пара 1↔12 / N↔13-N** → реверс решітки або тайбрейк рівного часу.

## Інваріанти логіки (вже з'ясовані на реальних даних)

- **Дублі фінішів у таблиці = легітимні** (двох поставили на одне місце бо заднього фолили і повернули). Імпортуємо.
- **Пропуски фінішів у таблиці = ігноруємо** (наслідок корекцій).
- **No-show** (0 валідних кіл ≥38с у гонці) → фініш порожній, 0 балів, старт зберігається. Наша логіка вже коректна. АЛЕ якщо пілот кваліфікувався і потрапив у групу, а на гонку не вийшов — він **займає стартовий слот** у нас, а таблиця його прибирає → зсув старту всіх за ним на +1 (не критично, на бали майже не впливає).
- **≥1 валідне коло = зарахований** (навіть 1 коло). Правила "75% кіл" НЕ існує.
- **Тайбрейк часу**: рівний best-lap до тисячних → кращий той, хто поставив коло РАНІШЕ (`byTimeThenTs` за `ts`). **Це свідоме рішення** — НЕ міняй на «за наступним колом» (пробували, відкотили: ризик зламати/сповільнити базовий scoring заради поодиноких кейсів).
- **Кількість квалі ≠ кількість race-груп.** LL часто має **4 квалі-групи**, що зливаються в **3 race-групи**. `filterPhases` ріже квалі-фази по `qualiCount`, race — по `groupCount` (фікс v0.3.36). Не «виправляй» 4 квалі на 3.
- **Розподіл по race-групах = чистий best-lap квалі** (`splitIntoGroups`): топ-12 → G1, наступні → G2 тощо. Таблиця робить так само. Зрідка організатор вручну пересаджує пілота в іншу групу всупереч часу — тоді застосовуй `edit.group` (`results.edits["Пілот|race"] = {group, startPos}`; поле `group` підтримується з v0.9.495; впливає на бали, стартова решітка інших НЕ перебудовується).
- **«Карт N» кола зливаються з іменем пілота** (`mergePilotNames` у scoring, v0.9.494): timing на старті сесії пише «Карт N» поки не підтягне ім'я → без злиття best-lap квалі неповний → неправильна група. Тепер scoring зливає автоматично (з reset-захистом: якщо `lap_number` скидається — це інший водій, НЕ зливаємо).
- **Колізії імен**: матчинг пілотів — за ПОВНИМ іменем (`buildNameMatcher`), не лише прізвищем (Синяговський Кирило/Віталій).
- **Дощ / крос-групові ротації** (рідко): у дощ організатор може переформувати групи Гонки 2 за абсолютним часом крос-групово (топ-4 з кожної групи → G1). Це складний edge-case — НЕ намагайся автоматизувати, познач користувачу і пропусти (напр. LL 19.05).

## Окремі скрипти (якщо треба точково)

| Скрипт | Призначення |
|---|---|
| `recreate.ts <id> [--name=][--date=] [--apply]` | перестворити з першого заїзду + валідація структури |
| `compare.ts <id> <gid/url>` | повна звірка наш standings vs таблиця |
| `diff-positions.ts <id> <gid/url>` | лише позиційні діфи (старт/фініш) по гонках |
| `import-penalties.ts <id> <gid/url> [--apply]` | штрафи з таблиці |
| `import-finishes.ts <id> <gid/url> [--apply] [--max=N]` | повернені фініші (sheet<ours, Δ≤N) + детект старт-аномалій |
| `import-starts.ts <id> <gid/url> <race> <group> [--apply]` | ручні старти для конкретної групи (рідко — переставлені організатором новачки) |
| `rebuild-editlog.ts <id> [--apply]` | відновити журнал змін з edits |

## Робочий процес із користувачем

1. Запусти **dry-run** `audit-competition.ts <id>`, покажи таблицю балів + стартові розбіжності.
2. Якщо лінкування зламане (відсутні гонки/квалі, DROPPED) → `--recreate` (див. «Повний flow» вище).
3. Стартові розбіжності — **розслідуй по джерелу** (таблиці стартів у звіті: гр-табл vs гр-наш, Кв-час). Більшість = граничні межі груп / no-show / «Карт N» / ручні перекиди організатора. Це НЕ завжди баг логіки. Не застосовуй наосліп.
4. БАЛИ/ФІНІШІ "таблиця опустила" — лишаємо наше (не баг). Спірні великі діфи (Δ>3) — показуй користувачу.
5. `--apply` після підтвердження. Видай фінальний звіт (таблиця балів `N/M` + стартові діфи + ручні зміни).

## НЕ роби

- НЕ підганяй наші фініші під таблицю якщо таблиця ОПУСТИЛА пілота (тільки повернення позицій).
- НЕ «виправляй» кількість квалі (4 квалі при 3 race-групах — це норма).
- НЕ міняй тайбрейк `byTimeThenTs` (за ts) — це свідоме рішення, відкат уже був.
- НЕ намагайся автоматизувати дощові крос-групові ротації — познач і пропусти.
- НЕ застосовуй на проді без dry-run і підтвердження.
- НЕ міняй `scoring.ts` щоб "підігнати" одне змагання — спершу переконайся що це системний баг на кількох. Базовий scoring крихкий — обговорюй зміни з користувачем.
- Backup БД — завжди лишай ТІЛЬКИ 2 останніх (див. Передумови).
